# Hardening backlog — vite-plugin-stylewright

> Generated by an automated multi-agent audit (5 finder dimensions, each finding adversarially verified). DEV-ONLY plugin: it runs only in `vite dev` and writes CSS back into `.svelte` source files. Threat model: the risk is anything OTHER than the developer's own page reaching/abusing the `/__stylewright/*` endpoints, or a bad path/payload slipping past the guards.
>
> Severities: P0 = must fix before release (exploitable / data-loss). P1 = serious. P2 = should-fix. P3 = minor.
> Workflow: fix highest-severity `TODO` items first; flip Status to `DONE` (or `WONTFIX` + reason) as you go.

## Security

### [SEC-1] (P0) CSRF: file-writing POST endpoints have no Origin/Host/token check — Status: DONE
- **Where:** `src/server/middleware.ts:174-296` (POST `/style`:229, `/apply`:251, `/edit`:273; `readBody`+`JSON.parse` at 232/254/276; `writeFile` at 243/265/287). *(reported by 2 finders — high confidence)*
- **Why it matters:** `createStylewrightMiddleware` mounts `/__stylewright/{edit,style,apply}` on the Vite dev server and each writes the developer's `.svelte` source to disk, with no Origin/Referer/Host/CSRF-token check anywhere in `src/server`. Because the body is `JSON.parse`d regardless of Content-Type, a cross-origin page can send a CORS "simple" request (text/plain JSON body, no preflight); the write side effect lands even though the attacker can't read the opaque response, and Vite's `allowedHosts`/Host check doesn't help (a localhost CSRF carries a legitimate Host). `resolveSvelteFile` only restricts to an existing in-root `.svelte`, so guessable defaults (`src/App.svelte`, `src/routes/+page.svelte`) are reachable and `/apply`'s create-path appends rules blindly — silent remote overwrite of source from any site a developer visits while `vite dev` runs. (Minor: the `/apply` `JSON.parse` is line 254, not 253.)
- **Fix:** Before any state-changing request, reject unless Origin/Referer exactly matches the dev-server origin and/or require a per-session secret token; also require `Content-Type: application/json` to force a CORS preflight.

### [SEC-2] (P0) `</style>` breakout via raw write in /style injects markup/script into source — Status: DONE
- **Where:** `src/server/patch.ts:311-346` (`applyStyleBlock`; raw write at 343-344; `isCompleteCss` guard 291-303). Live via `src/server/middleware.ts:229-248` and client `saveCss` (`src/client/index.ts:30-34`). *(reported by 2 finders — high confidence)*
- **Why it matters:** `applyStyleBlock` writes client CSS verbatim via `ms.overwrite(block.start, block.end, css)` — no postcss re-serialization. The only guard, `isCompleteCss`, just checks the CSS parses with non-empty values, so a comment/`url()`/`content` value containing a literal `</style>` passes and is written inside the component's `<style>` block (reproduced). When Svelte recompiles, the HTML tokenizer closes the style block at that literal tag. Mechanism correction: a `<script>` payload most likely triggers a Svelte duplicate-top-level-script compile error (DoS); reliable execution in the dev origin comes from an attribute-handler markup payload such as `</style><img src=x onerror=...>`. The `/edit` and `/apply` paths serialize via postcss `root.toString()`, which escapes `</style` to `\3c /style` (empirically verified — though one finder flagged uncertainty about `/apply`), so the defect is specific to the raw-write `/style` path. P0 is conditional on the SEC-1 CSRF chain (standalone same-origin it's ~P2); SEC-1 is corroborated, so chained this becomes code execution in the dev origin plus exfiltration of every component via a callback into `/__stylewright`.
- **Fix:** Don't write client CSS raw — re-serialize through postcss (`postcss.parse(css).toString()`), or reject any css containing `</style`/`</script`/`<!--` (case-insensitive) before writing. Applying the same re-serialization to `/apply` is safe regardless.

## Correctness

### [COR-1] (P1; auditor said P2, verifier upgraded) HSL parsing maps hue 240–300 to the wrong RGB sextant — Status: DONE
- **Where:** `src/client/color.ts:78`
- **Why it matters:** `parseColor`'s HSL→RGB sextant for 240 ≤ hue < 300 uses `rr = cc; bb = x;` (C,0,X) — byte-identical to the 300–360 branch — instead of the correct `rr = x; bb = cc;` (X,0,C). So any `hsl()` value with hue in [240,300) (blues/indigos/violets) parses to the wrong color: `hsl(240,100%,50%)` (pure blue) → RGB(255,0,0) (pure red). Because `openColor` seeds the picker from this parse and the format-cycle / picker-drag re-emit `formatColor()` from the wrong h/s/v, a routine interaction silently rewrites the source value to a wrong color — silent data corruption is why it was upgraded to P1. (Correction: the inline token swatch uses the raw author string and renders correctly; only the picker's preview swatch, the dedup/equality, and the source rewrite are wrong.)
- **Fix:** Change line 78 to `else if (hh < 300) { rr = x; bb = cc; }`.

### [COR-2] (P1; auditor said P2, verifier upgraded) Undo/redo after a structural op can patch the WRONG source rule — Status: DONE
- **Where:** `src/client/panel.ts:527-531` (`applyHistState`), `502-510` (`removeRule`); `src/server/patch.ts:160-201`
- **Why it matters:** History snapshots clone `Rule[]` carrying the server's walk-order ordinal `id`s, and `applyHistState` restores a snapshot then unconditionally `save()`s → POST `/apply`, which patches rules to source nodes purely by that numeric id. But ids shift whenever a rule is created/removed, so after `removeRule()`+`reloadRules()` an undo to the pre-remove snapshot sends stale ids: `idMap.get(staleId)` resolves to a DIFFERENT current rule and `reconcileDecls` overwrites its declarations with the snapshot rule's decls (selector kept, body swapped) — silent cross-rule corruption. Symmetrically the deleted rule isn't recreated (it has an id, so it's a patch target not a create) and undo of `addOverride` sends no `removeIds`, so the in-memory view and on-disk source diverge. Upgraded to P1: the ordinary "delete a rule, then Ctrl+Z" workflow silently corrupts source (borders P0; kept P1 as dev-only and recoverable via VCS).
- **Fix:** Re-key by stable identity (selector + media path) before replaying a restored snapshot, or record structural ops as inverse operations; at minimum mark pre-structural-op history entries as non-replayable via `/apply`.

### [COR-3] (P2) Slow-path reconcile destroys in-rule comments — Status: TODO
- **Where:** `src/server/patch.ts:274` (`reconcileDecls` slow path)
- **Why it matters:** When a save adds/removes/reorders a declaration (the fast-path equality at 263 fails), `reconcileDecls` calls `node.removeAll()` then re-appends only the decls. `removeAll()` deletes ALL child nodes including comment nodes, and the re-appended decls get default postcss raws, so an inline comment inside the rule is lost (reproduced with the project's postcss). `readRules` only surfaces decl nodes, so the client can't round-trip the comment back. Corrections to the original report: untouched declarations are NOT generally reindented (tab indentation is preserved in normal multi-rule blocks; reindent only happens when the edited rule is the sole one with decls), and the slow path's "rebuilds the rule body" behavior is already documented at `patch.ts:250-256` — the comment loss is the real, undocumented consequence.
- **Fix:** Diff/splice decl nodes in place (insert/remove/reorder only changed decls) instead of `removeAll()`, preserving comment siblings.

### [COR-4] (P2) readBody corrupts multi-byte UTF-8 in save bodies — Status: TODO
- **Where:** `src/server/middleware.ts:57-61`
- **Why it matters:** `readBody` does `let data=''; req.on('data', chunk => data += chunk)`, coercing each Buffer chunk to a UTF-8 string independently, so a multi-byte character that straddles a chunk boundary decodes to replacement characters. Bodies up to 1MB arrive in ~16-64KB chunks, so large/non-ASCII `<style>` or JSON payloads (`content:"…"`, non-ASCII font names) sent to `/style`, `/apply`, `/edit` can corrupt. Correction: the dominant outcome is NOT a thrown `JSON.parse` — U+FFFD is valid inside a JSON string and corruption only hits multibyte chars inside the quoted CSS value, so `JSON.parse` succeeds and silent mojibake is written to the developer's source (the worse case).
- **Fix:** Collect Buffers and decode once: `const chunks=[]; req.on('data',c=>chunks.push(c)); req.on('end',()=>resolve(Buffer.concat(chunks).toString('utf8')))`, enforcing the size cap on total byte length (see TEST-6).

### [COR-5] (P3) Breakpoint rename targets the wrong @media for nested at-rule chains — Status: TODO
- **Where:** `src/client/panel.ts:514-523` (`commitBreakpoint`) vs `src/server/patch.ts:172-178` (`mediaRenames`)
- **Why it matters:** `commitBreakpoint` computes the new params from `r.media[0]` (the OUTERMOST at-rule) and sends `mediaRenames:[{id,params}]`, but the server renames `node.parent` (the INNERMOST at-rule). For a single `@media` these coincide; for a nested chain (e.g. a width `@media` wrapping an orientation `@media`), the chip shows as editable (media[0] is width-based) but committing overwrites the inner media's params with the width string, dropping the orientation constraint. If the inner at-rule is `@supports` the rename silently no-ops (the guard requires `at.name === 'media'`). Rare in component `<style>`, hence P3, though it silently loses a media constraint in source.
- **Fix:** Rename the specific at-rule node the client edited (pass the chain depth, or have the server walk up to the matching width `@media`) instead of always `node.parent`.

### [COR-6] (P3) isColorValue accepts invalid 5- and 7-digit hex — Status: TODO
- **Where:** `src/client/color.ts:49`
- **Why it matters:** `/^#([0-9a-f]{3,8})$/i` matches hex of length 3–8 inclusive, but only 3/4/6/8 are valid CSS hex colors. A value like `#12345` or `#1234567` is classified as a color, opening the picker and routing through `parseColor`, which slices it blindly and yields a garbage color rather than treating it as plain text. Impact is minor: it only triggers on a hex literal that is already invalid CSS (a typo in the developer's own source); no crash, no data loss.
- **Fix:** Restrict to exact lengths, e.g. `/^#([0-9a-f]{3}|[0-9a-f]{4}|[0-9a-f]{6}|[0-9a-f]{8})$/i`.

## Performance

### [PERF-1] (P2; auditor said P1, verifier downgraded) orderedView() walks the whole page DOM on every render — Status: TODO
- **Where:** `src/client/panel.ts:1185-1203` (called unconditionally at 1325)
- **Why it matters:** `buildEditor()` calls `orderedView()` on every editor render. `orderedView` builds a Map of EVERY element via `Array.from(document.querySelectorAll('*'))` plus one `document.querySelector` per rule, and `render()` runs on every CSS keystroke / number-scrub / slider drag. The DOM ordering can't change from a CSS edit, so this is pure waste on the hot path. Downgraded to P2: `render()` already tears down and rebuilds the entire overlay each keystroke (see PERF-2), so that rebuild — not `orderedView` — is the dominant per-keystroke cost, and R (rules in one component) is small; the order is invariant under edits so memoization is valid but won't alone remove typing lag.
- **Fix:** Memoize the ordering (and element Map) keyed by a stable revision (pick/reload/resize), invalidating only when the live DOM actually changes.

### [PERF-2] (P2) Entire overlay DOM is torn down and rebuilt on every keystroke — Status: TODO
- **Where:** `src/client/panel.ts:281-304`
- **Why it matters:** `render()` does `clear(this.rootEl)` then `appendChild(this.buildOverlay())`, recreating the header, breadcrumb, every rule and every declaration input from scratch on each state change. Each keystroke also runs `setRules` → `cloneRules()` (a full model deep copy) and `buildEditor` re-tokenizes every non-focused declaration value; the caret save/restore machinery exists precisely because the focused input is destroyed and rebuilt each keystroke. This is the amplifier that makes the other per-render costs fire per keystroke. Corrections: the FOCUSED value field uses `rawValueInput`, so it is NOT re-tokenized while typing (only the other decls are), and the input-lag/GC-churn magnitude is unverified — sub-frame for normal components, only plausible on pathologically large style blocks.
- **Fix:** Avoid full teardown on value edits — while an input is focused and only its value changed, let the native input hold the text and skip re-render (re-tokenize on blur, as `rawValueInput` already does); or diff/patch instead of `clear()`+rebuild, and drop the per-keystroke `cloneRules` for a targeted copy of the edited rule.

### [PERF-3] (P3; auditor said P1, verifier downgraded) overriddenBy() rescans all rules per declaration on every render — Status: TODO
- **Where:** `src/client/panel.ts:1159-1180`
- **Why it matters:** `declLine()` calls `overriddenBy(ri, d.p)` for every declaration of every displayed rule each render; `overriddenBy` iterates the full rules array computing `normSel(r.sel)` (a regex) per rule, giving O(decls × rules) regex normalizations per render, plus uncached `window.matchMedia(params)` calls. The mechanism is confirmed but the impact was heavily overstated: `state.rules` holds only ONE component's `<style>` block (a handful to a couple dozen rules, not a 400-decl design system), `matchMedia` is gated behind a same-selector short-circuit, and ~16k `normSel` runs is ~1-3 ms — sub-millisecond at realistic sizes and dominated by `orderedView`/the full rebuild anyway. Hence a P3 micro-opt.
- **Fix:** Precompute once per render a `Map<normalizedSelector, ruleIndex[]>` and memoize `ruleActive`/`matchMedia` keyed by the params string.

### [PERF-4] (P3; auditor said P2, verifier downgraded) DOM-tree pane re-walks the whole document on every expand/collapse — Status: TODO
- **Where:** `src/client/panel.ts:1523-1558`
- **Why it matters:** `buildTree()` memoizes by `state.treeRev`, but `toggleNode()` calls `bumpTree()` which increments `treeRev`, invalidating the cache, so each expand/collapse (and each pick) triggers a full `buildDomTree(document.body)` walk plus row reconstruction even though only one subtree's visibility changed. Downgraded to P3: the DOM pane is off by default and gated behind an explicit toggle, the toggle already incurs a full overlay re-render regardless, the row reconstruction is unavoidable when the visible set changes, and the only avoidable work (the O(n) model walk) is a few ms on typical pages — the "sluggish on large DOM" claim is unmeasured.
- **Fix:** Cache the `buildDomTree` model separately from visibility/selection state; on expand/collapse re-render rows from the cached model instead of bumping `treeRev` and re-walking the document.

## Best practices / packaging

### [PRAC-1] (P2) package.json missing repository / bugs / homepage metadata — Status: TODO
- **Where:** `package.json:1-16`
- **Why it matters:** This is a 0.0.1 staged for first npm publish (`prepublishOnly` build, `files` allowlist), and the README advertises the GitHub repo, but `package.json` has no `repository`, `bugs`, or `homepage` field. The npm page will have no source link, `npm repo`/`npm bugs` won't work, and issue routing/provenance is lost — standard release hygiene for a public tool soliciting issues/PRs. (Purely metadata; zero functional/security impact; sits at the P2/P3 boundary.)
- **Fix:** Add `"repository": { "type": "git", "url": "git+https://github.com/Greg-J/vite-plugin-stylewright.git" }`, `"bugs": { "url": ".../issues" }`, and `"homepage"`.

### [PRAC-2] (P3; auditor said P2, verifier downgraded) exports map types not split per-condition; orphaned dist/index.d.cts — Status: TODO
- **Where:** `package.json:17-23`
- **Why it matters:** The package is `"type":"module"` so `dist/index.d.ts` is treated as ESM types, but the exports map exposes a CJS `"require":"./dist/index.cjs"` with only a single top-level `"types":"./dist/index.d.ts"`. Under `moduleResolution` node16/nodenext a `require()` consumer resolves ESM-flavored types for a CJS module (the FalseESM hazard `attw` flags); tsup already emits `dist/index.d.cts` for this case but nothing references it. Downgraded to P3: it's a Vite plugin consumed almost exclusively via `import` under `moduleResolution:"bundler"` (which doesn't differentiate the conditions and `attw` doesn't flag), the `.d.cts` is byte-identical and `dist/index.cjs` already emits `__esModule`+`default`, so the worst case is a benign `attw` warning for an uncommon CJS/node16 consumer.
- **Fix:** Nest types per condition — `".": { "import": { "types": "./dist/index.d.ts", "default": "./dist/index.js" }, "require": { "types": "./dist/index.d.cts", "default": "./dist/index.cjs" } }`; validate with `npx @arethetypeswrong/cli`.

## Test gaps

### [TEST-1] (P2; auditor said P1, verifier downgraded) POST /edit does not validate value — writes 'undefined' / allows extra-rule injection — Status: TODO
- **Where:** `src/server/middleware.ts:280-283`; `src/server/patch.ts:98-102`
- **Why it matters:** The `/edit` guard only checks `!edit.selector || !edit.prop` — value is never validated and `/edit` has no `isCompleteCss` guard (unlike `/style`). When the property exists, `applyEdit` does `decl.value = edit.value` with value `undefined`; postcss doesn't throw and serializes `.btn { color: undefined; }`, which is written to source and breaks the Svelte compile (the append branch instead throws → 500, no write). A crafted value like `red } .evil{display:none` serializes an extra rule. Downgraded to P2: triggering needs a malformed/CSRF request (the real overlay always sends a string), and it's no escalation — anyone who can reach `/edit` can already write an entire arbitrary CSS block via `/style` or `/apply`; writes stay confined to in-root `.svelte` and the break is a recoverable single-file compile error. Defense-in-depth plus missing tests, not a release-blocker.
- **Fix:** Require `typeof edit.value === 'string'` and run `isCompleteCss` on the result before `writeFile` (mirroring `/style`); add tests for missing/non-string value (no write / 400) and for a value containing `}` (no rule breakout).

### [TEST-2] (P2) Entire createStylewrightMiddleware HTTP contract is untested — Status: TODO
- **Where:** `src/server/middleware.ts:174-296`
- **Why it matters:** Only `createHtmlInjectMiddleware` has tests (`inject.test.ts`); the read/write API (GET `/rules`, GET/POST `/style`, POST `/apply`, POST `/edit`, GET `/client.js`) has no coverage of method routing or status contract (404 on failed `resolveSvelteFile`, 400 on invalid JSON / bad request shape, 500 on read/write failure, 200 success shapes). A routing-order or wrong-status regression on the handlers that perform disk writes would go unnoticed. Somewhat over-broad — the transform/write logic is already unit-tested in `patch.test.ts` — but the single highest-value gap is the `resolveSvelteFile` path-escape → 404 test.
- **Fix:** Drive each endpoint with a fake req/res harness (or a real `http.Server` + fetch): assert GET `/rules?file=../escape` → 404, POST `/style` non-JSON → 400, POST `/style` with `css:123` / POST `/apply` with `rules:'x'` → 400, POST `/edit` valid → 200 + file changed, and a GET on a POST-only route falls through.

### [TEST-3] (P3; auditor said P1, verifier downgraded) Security guard resolveSvelteFile has no adversarial-path tests — Status: TODO
- **Where:** `src/server/middleware.ts:32-45`
- **Why it matters:** `resolveSvelteFile` is the only thing keeping requests inside the project root, yet it is module-local and never exercised by any test. Containment is a hand-rolled lowercased string-prefix check plus a trailing-sep append. Downgraded from P1: the current guard is actually correct for the realistic threat model — the `/proj-evil` sibling escape is blocked today by the trailing-sep append (only a hypothetical future refactor reopens it), percent-encoded `..%2f` traversal is decoded by `URLSearchParams` then caught by `normalize`+`startsWith`, and the only genuine weakness (POSIX case-fold over-match on a case-variant sibling `.svelte`) is extraordinarily improbable. It's a test-coverage gap on a currently-correct guard, not a live defect.
- **Fix:** Add a thin test export and a table-driven test asserting `null` for `../../../etc/passwd`, an out-of-root absolute path, a prefix-sibling (`/projEVIL/x.svelte` vs `/proj`), a POSIX case-variant sibling, percent-encoded traversal, and a non-`.svelte`/non-existent path — and the abs path for a legit in-root `.svelte`.

### [TEST-4] (P3; auditor said P2, verifier downgraded) findStyleBlock first-match regex can grab <style> inside script strings/comments — Status: TODO
- **Where:** `src/server/locate.ts:18`
- **Why it matters:** `STYLE_RE` is a non-anchored first-match regex over the whole file, so a `<style>…</style>` literal embedded in a `<script>` string before the real block wins (reproduced: inner=`'FAKE'`). On such a component, ops are misdirected and `applyStyleBlock` (whose guard validates the NEW css, not the target) could overwrite the wrong region. Downgraded to P3: the flagship `'FAKE'` case does NOT corrupt via `readRules`/`applyEdit` (`postcss.parse('FAKE')` throws → `rules:[]`, `applyEdit` no-ops); corruption needs the embedded string to be valid CSS, and the code editor first shows `readStyle`'s wrong region to the developer (visible, not silent). Extremely rare input (a literal `<style>` in a script string before the real block), developer's own recoverable source, no security boundary.
- **Fix:** Add locate tests (script-string `<style>`, commented-out `<style>`, two blocks) and harden the matcher to skip `<script>…</script>` / comment regions (or scan from the last `</style>`).

### [TEST-5] (P3; auditor said P2, verifier downgraded) applyRules untested for non-media at-rules, nested chains, decl reordering — Status: TODO
- **Where:** `src/server/patch.ts:188-192`
- **Why it matters:** `applyRules` tests cover only `@media`/`@keyframes`. Gaps: `@font-face`/`@supports`/`@import` preservation through a patch is unverified; prune-on-remove only checks `node.parent`, so removing the last rule from `@supports{@media{…}}` prunes the inner `@media` but leaves an empty `@supports`; multi-level media chains on `insertRule` are only tested single-level; and the slow rebuild path drops comments interleaved between decls on reorder. Downgraded to P3: none break Svelte compilation or lose CSS values — an orphaned empty `@supports{}` is valid CSS, comment loss is only on the slow path, and `@font-face`/`@import` ride the same proven "serialize untouched node" path. The worthwhile additions are the nested-prune and interleaved-comment cases.
- **Fix:** Add `applyRules` tests for `@font-face`/`@supports` survival through an edit, nested `@supports>@media` prune behavior, a two-level media chain create, and a decl reorder (documenting comment behavior).

### [TEST-6] (P3) readBody 1MB cap untested and measured in code units, not bytes — Status: TODO
- **Where:** `src/server/middleware.ts:55-65`
- **Why it matters:** The only body DoS guard is `data.length > 1_000_000`, but `data` is a JS string so `.length` counts UTF-16 code units, not bytes (the guard runs in the *looser* direction — ~3MB of CJK bytes pass), and on reject the request stream is never destroyed. No test asserts the cap fires or that an oversized POST yields 400 with no write. Worst case is a recoverable dev-server memory spike, hence P3.
- **Fix:** Add a middleware test streaming a >1MB body to POST `/edit` asserting 400 and no file write; optionally measure `Buffer.byteLength` and call `req.destroy()` on overflow.

## Appendix — refuted during verification (not actionable)

- `resolveSvelteFile uses path.normalize, not realpath — symlink/junction escape` (security) — reads/writes only touch CSS inside a pre-existing `<style>` block (`findStyleBlock` gates all paths), cross-origin read is SOP-blocked (no CORS), and planting an in-root `.svelte` symlink already implies equal-or-greater dev-time RCE; at most P3 hardening.
- `Request body cap does not bound memory — rejects at 1MB but keeps buffering` (security) — empirically TCP backpressure caps buffering at ~cap + one in-flight window (~2MB) once the 400 is sent; no unbounded growth. (Residual: not calling `req.destroy()` leaves a stalled ~2MB connection until keep-alive timeout — P3 hygiene.)
- `500 responses leak absolute filesystem paths via String(err)` (security) — no CORS so unreadable cross-origin; to the same-origin dev page it discloses nothing Vite doesn't already via `/@fs/` URLs, the HMR overlay, and source maps. Cosmetic at most.
- `No concurrency control on read-modify-write of source files` (security) — the only client debounces and POSTs the full superset rule model, so interleaved writes are idempotent/superset-preserving; a mutex gives zero CSRF mitigation and the `/edit` endpoint is unused by the panel.
- `History.record() JSON.stringify's the entire rules model on every render` (performance) — the rules model is one small component's `<style>` block; the stringify is negligible next to the full DOM teardown+rebuild and the `cloneRules` already done per keystroke. At most a P3 revision-counter micro-opt.
- `pageColors() re-tokenizes every declaration per render/drag frame; loadedFonts() rebuilds+sorts per render` (performance) — bounded small data sets, `tokenize` is one cheap regex pass, strictly dominated by the unconditional full-overlay rebuild (which already tokenizes every decl); caching wouldn't perceptibly change frame time.
- `Middleware reads the client bundle from disk synchronously on every GET /client.js` (performance) — auditor's own judgment is "negligible / not required for correctness"; sub-ms read on an infrequent `defer`-script load, and the fresh re-read is a deliberate, documented "rebuild shows on refresh" tradeoff.
- `No engines.node despite Node-only APIs and node18 build target` (practices) — `node:` specifiers and `fs.promises` run on Node 14/16 (node18 is a transpile target, not an API floor), and the Node-18 floor is already implied transitively by the declared `vite>=5` peer dependency.
- `exports map seals ./package.json subpath access` (practices) — the plugin is only ever a default import; Vite orders plugins via `enforce`, not package.json reads, and nothing in the repo deep-imports it. No concrete breakage.
- `HTML-inject middleware swallows all injection errors with no diagnostic` (practices) — the cited "sibling pattern" (`loadClient`'s `console.warn`) is a browser-served JS string, not a server log; no server logging exists anywhere and the guarded ops can't realistically throw. Cosmetic nicety.
- `Raw error strings (absolute paths) returned to the browser in 5xx responses` (practices) — duplicate of the security `String(err)` finding above; the disclosed path is a guard-validated in-project `.svelte` the same-origin caller already supplied, unreadable cross-origin.
- `Containment guard does not resolve symlinks — no realpath test, latent escape` (testgaps) — duplicate of the security symlink finding; misclassified as a test gap (the proposed test can't pass without first adding the check), endpoints return only `<style>`-block CSS (no raw bytes / SSH keys), and there's no remote vector to plant the symlink. P3 hardening at most.
